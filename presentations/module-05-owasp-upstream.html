🌍 What is OWASP?

Open Worldwide Application Security Project

A non-profit, vendor-neutral community producing free resources, tools, and training for application security. Local chapters meet worldwide.

  • Top 10 — awareness document, ranked categories of web-app risk
  • ASVS — Application Security Verification Standard (a checklist by maturity level)
  • SAMM — Software Assurance Maturity Model
  • Cheat Sheets — concise per-topic guidance
  • Tools — ZAP, Dependency-Check, Juice Shop, ModSecurity CRS
If learners take one bookmark away, it should be cheatsheetseries.owasp.org.

🔒 A01 — Broken Access Control

The user accesses something they shouldn't

Includes IDOR, missing function-level checks, JWT tampering, force-browsing.

  • Smell: /api/users/123/orders — can I change 123 to 124?
  • Fix: deny by default; check authorisation server-side on every request
  • Test: swap session tokens, change IDs, fuzz HTTP verbs
A01 is consistently the highest-impact and most under-tested category. Devs assume "the front-end won't show that button" — attackers don't use the front-end.

🔓 A02 — Cryptographic Failures

  • Weak hashes (MD5, SHA-1) for passwords
  • Hard-coded keys in source / images
  • Plaintext data in transit (no TLS) or at rest (unencrypted volumes)
  • Weak randomness (Math.random() for tokens)
Default: use platform crypto (libsodium, the language's secrets / crypto.randomBytes) and managed Key Vault. Don't roll your own.
"Don't roll your own crypto" is the single most repeated security rule for a reason. Make peace with it.

💉 A03 — Injection

Forms

  • SQL / NoSQL injection
  • Command injection
  • LDAP / XPath injection
  • Server-side template injection
  • Reflected / stored / DOM XSS

Fix pattern

  • Parameterised APIs — bind, don't concatenate
  • Output encoding by context (HTML/attr/JS/URL)
  • Allow-list validation
  • Use ORMs & safe shell APIs
XSS lives here in 2021. Same root cause as SQL injection: data treated as code.

📝 A04 — Insecure Design

The bug is in the architecture, not the code

Examples: a "forgot password" flow that emails the password; a public download URL with no rate limit; OTP that doesn't expire.

  • You can't patch your way out — needs design review & threat modelling
  • Connects directly to module 4 — STRIDE catches A04 issues
A04 is the explicit reward for doing threat modelling. Tools cannot find these by themselves.

⚙️ A05 — Security Misconfiguration

  • Default admin credentials still in place
  • Verbose error pages leaking stack traces
  • CORS * with credentials
  • Cloud-storage buckets accidentally public
  • Unhardened container images, kernel sysctls left default
Detect with: IaC scanners (Checkov, tfsec), CIS benchmarks, Defender for Cloud recommendations.
Connects to module 3 — most "infrastructure threats" are A05.

📦 A06 — Vulnerable & Outdated Components

  • Direct & transitive dependencies with known CVEs
  • Outdated base images (ubuntu:18.04)
  • Abandoned libraries
Detect with SCA tools: Dependabot, Renovate, Snyk, OWASP Dependency-Check, Trivy. Generate & sign an SBOM (CycloneDX or SPDX).
Why it matters: Log4Shell (Dec 2021) was a single Java logging library compromise that touched millions of apps overnight.
SBOM is becoming a regulatory expectation (US Executive Order 14028). Worth knowing the term.

👤 A07 — ID & Authentication Failures

  • Weak password policy / credential stuffing
  • No MFA on privileged accounts
  • Session fixation, predictable session IDs
  • Token never expires; logout doesn't actually invalidate
  • Recovery flows that leak account existence
Default to identity providers: IAM Identity Center, Auth0, Okta, Cognito. Resist the urge to build your own login.
If your team is still building login screens from scratch, that's a strong refactor candidate.

🔗 A08 — Software & Data Integrity Failures

  • Unsigned package downloads (curl | bash)
  • Auto-update from untrusted source
  • Insecure deserialisation
  • Unverified CI/CD plugins, untrusted GitHub Actions
Mitigations: sign artefacts (cosign / Sigstore), use SLSA build levels, lock dependency versions, review third-party CI plugins.
The post-SolarWinds category. Supply-chain attacks live here.

📝 A09 — Logging & Monitoring Failures

  • No audit log for sensitive actions
  • Logs don't reach a SIEM / centralised store
  • Log fields swallow PII, can't be redacted later
  • No alerting on auth failure spikes
Sets up Module 8 — we'll do this properly with CloudWatch and KQL.
Why is logging in the Top 10? Because you can't respond to what you can't see.

😇 A10 — Server-Side Request Forgery (SSRF)

The server fetches a URL the attacker controls

Classic target: cloud metadata services (169.254.169.254) → steal instance credentials → pivot.

  • Smell: any feature that takes a URL parameter and fetches it server-side
  • Mitigations: allow-list outbound destinations, disable redirects, use IMDSv2, network egress filtering
SSRF graduated to its own slot in 2021 thanks to cloud-attack reality. Capital One breach is the reference incident.
