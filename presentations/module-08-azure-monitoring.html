🔍 Why monitor?

  • Detect — bugs & attacks the pipeline missed
  • Respond — turn signal into ticket / page / playbook
  • Forensic — reconstruct what happened, when, who
  • Compliance — auditors expect immutable, retained logs
  • Feedback to dev — close the loop with what's actually being attacked
Connection to A09: "Logging and monitoring failures" is in the OWASP Top 10 for a reason.
"You can't respond to what you can't see" — repeat this. Monitoring isn't a luxury; it's the only path to learning from production.

📋 Three pillars of telemetry

Metrics

Pre-aggregated time series. Cheap, fast.

Best for: "is it healthy now?" alerts.

Logs

Discrete events with structure.

Best for: "what happened?" investigations & security detections.

Traces

Request graphs across services.

Best for: latency root-causing.

Security work lives mostly in logs. KQL is the language; Log Analytics is the store.
For DevSecOps, metrics catch DoS-shaped issues; logs catch everything else.

💾 Log Analytics workspace

  • Central log store — per-tenant, multi-region
  • Resources emit logs via diagnostic settings
  • Tables: SigninLogs, AuditLogs, AzureActivity, AppRequests, AppDependencies, AzureDiagnostics…
  • Retention: 30 / 90 days default; archive up to years (cheaper)
Cost reality: ingestion is the biggest line item. Filter at source — don't ingest you-don't-need.
Telemetry cost surprises kill monitoring programs. Start narrow.

📝 KQL in 90 seconds

// All sign-ins for one user in the last day
SigninLogs
| where TimeGenerated > ago(1d)
| where UserPrincipalName == "alex@contoso.com"
| project TimeGenerated, ResultDescription, IPAddress, AppDisplayName
| order by TimeGenerated desc
  • Pipe-based, like a SQL/Splunk hybrid
  • Verbs: where, project, summarize, extend, join, render
  • Time helpers: ago(), now(), bin()
The same language is used in Sentinel, Defender XDR, and Application Insights. Worth investing in.

🔎 A useful detection — impossible-travel signal

SigninLogs
| where TimeGenerated > ago(1h)
| where ResultType == 0
| summarize Locations = make_set(Location), IPs = make_set(IPAddress) by UserPrincipalName, bin(TimeGenerated, 10m)
| where array_length(Locations) > 1
| project TimeGenerated, UserPrincipalName, Locations, IPs
What it catches: a user appearing successfully signed in from two countries within 10 minutes — almost always credential misuse.
Show this query running in the lab. Tweak the bin window if false positives are too high (VPN users, etc.).

🔔 Alerts & action groups

Alert rule = condition + action

  • Metric alert: threshold on a numeric series
  • Log alert: KQL query returns > 0 results
  • Activity-log alert: control-plane event match

Action groups

  • Email, SMS, voice
  • Webhook (Slack, Teams, PagerDuty, JIRA)
  • Azure Function / Logic App / Automation runbook
  • Event Hub / Event Grid
Pattern: high-severity → PagerDuty & ticket; medium → Slack channel; low → dashboard tile only.
Action-group reuse is key — define once, attach to many rules.

🛡️ Defender for Cloud

  • CSPM: continuous posture management — surfaces misconfig recommendations
  • CWPP: workload protection — threats on VMs, containers, App Service, SQL, Storage…
  • Secure Score: 0–100 measure of posture for KPIs
  • Findings & alerts flow into Log Analytics & Sentinel
Free tier gives you CSPM. Paid Defender plans add CWPP per resource type.
Many teams enable Defender, ignore Secure Score, and miss 80% of the value. Bake the score into a quarterly review.

🌝 Microsoft Sentinel

  • SIEM + SOAR built on Log Analytics — same workspace, same KQL
  • Connectors: 100+ for Azure, AWS, GCP, on-prem, SaaS
  • Analytics rules: scheduled, NRT, ML, Microsoft-managed
  • Workbooks & hunting: operational + threat-hunting UIs
  • Playbooks via Logic Apps for automated response
Decision aid: Defender for Cloud finds what's wrong now; Sentinel detects what's happening over time.
Don't pretend learners need to operate Sentinel after this course. Frame it as "this is where you'd buy in if you grow past Azure Monitor + Defender."

🎯 Detection use cases worth wiring up day one

  1. Repeated failed sign-ins for one user (brute force)
  2. Successful sign-in from impossible travel
  3. Privilege role assignment (e.g. Owner) on a subscription
  4. NSG rule changed to allow 0.0.0.0/0 on management ports
  5. Storage account / Key Vault firewall opened to public
  6. Application logs — spike in 5xx after deployment
These six catch a surprising fraction of real incidents. Build them, tune them, then expand.
