🧠 Why automate security?

  • Consistency — same check on every change, no exceptions
  • Speed — minutes per build, not days per audit
  • Forcing function — what's in CI gets fixed; what's in a PDF doesn't
  • Evidence by-product — every run is an audit trail
  • Scale — humans can't review thousands of dependencies
Compounding rule: the earlier in the pipeline a check runs, the more times the cost is paid back.
Recap: this is the operational form of "shift left" from module 1.

📝 Anatomy of a security-aware pipeline

pipeline {
  agent any
  stages {
    stage('Build')      { steps { sh './gradlew assemble' } }
    stage('SAST')       { steps { sh 'semgrep --config auto --error' } }
    stage('SCA')        { steps { sh 'trivy fs --severity HIGH,CRITICAL --exit-code 1 .' } }
    stage('Image scan') { steps { sh 'trivy image --severity CRITICAL --exit-code 1 app:$TAG' } }
    stage('DAST (ZAP)') { steps { sh 'zap-baseline.py -t http://app.dev -I' } }
  }
}
Walk top-to-bottom on the slide. Each stage maps to a previous module — call them out as you go. Real pipelines also have unit tests, deploy stages, and post-actions; trimmed here for clarity.

🦎 Integrating OWASP ZAP

stage('DAST — ZAP baseline') {
  steps {
    sh '''docker run --rm --network ci-net \
            -v $WORKSPACE:/zap/wrk:rw \
            ghcr.io/zaproxy/zaproxy:stable \
            zap-baseline.py -t http://app.dev:8080 \
              -r zap-report.html -J zap-report.json -I'''
  }
  post {
    always { publishHTML target: [reportFiles: 'zap-report.html', ...] }
  }
}
Pattern: baseline on every PR · full scan nightly against staging · use -I initially to ingest reports without failing builds; tighten thresholds after triage.
Day-one mistake is failing builds on day one. Triage first, gate second.

💤 What fails the build?

SeverityDefault policy
Critical (RCE, auth bypass, leaked secret)Block, no exceptions
HighBlock; bypass requires approval recorded as artefact
MediumWarn; track in backlog with SLA
Low / informationalReport only; no block
Path to green: every blocking finding must come with a documented remediation step. "Just fix it" isn't a path.
Without a path to green, gates breed bypasses. Treat the gate as a contract, not a firewall.

🔥 Pipeline anti-patterns

  • Allow-list everything to "fix" the noise — you've turned off the tool
  • Run scans only on main — PRs ship findings; main inherits them
  • One mega-stage — failures swallow the actual reason
  • Tools that don't surface in the PR — if it's not in the diff, devs don't see it
  • "Runs nightly only" — problem found 18 hours after merge isn't shift-left
Pattern: keep the dev's feedback loop tight. Anything that breaks that breaks DevSecOps.
