🔎 What is penetration testing?

An authorised, simulated attack

A time-boxed engagement where a tester attempts to compromise a defined target the way a real adversary would — documenting each step and impact.

  • Goal: demonstrate exploitable risk, not just "we have a finding"
  • Output: a report with reproducible findings + remediation guidance
  • Authority: always written, always specific
"Authorised" is doing all the work in that definition. Pen testing without scope is cracking.

🏫 Black, grey, white box

Black box

Tester gets nothing — only the target's name.

  • Realistic external view
  • Slow start

Grey box

Tester gets some access — user creds, basic docs.

  • Best ROI for most apps
  • Realistic insider

White box

Tester gets source, IaC, architecture.

  • Deepest coverage
  • Best for security-critical code
Default for app pen tests: grey box with named accounts at each role.
Choose based on what you're trying to learn. Black-box "is anyone exploiting our public surface?"; white-box "is the design sound?"

📚 Methodologies you'll cite

  • PTES — Penetration Testing Execution Standard. Seven phases. Industry default.
  • OSSTMM — Open Source Security Testing Methodology Manual. Strong on metrics.
  • OWASP WSTG — Web Security Testing Guide. Web-app specific.
  • NIST SP 800-115 — pragmatic federal guidance.
  • MITRE ATT&CK — not a methodology, a coverage matrix.
Every methodology agrees on the broad shape. Don't get religious. Pick PTES + OWASP WSTG for web apps.

🔐 PTES — the seven phases

#PhaseWhat happens
1Pre-engagementScope, RoE, contracts, contacts
2Intelligence gatheringOSINT, recon (passive & active)
3Threat modellingMap findings → likely attack paths
4Vulnerability analysisIdentify exploitable weaknesses
5ExploitationGain access; capture proof
6Post-exploitationPrivilege escalation, pivot, persist (in-scope)
7ReportingFindings, impact, remediation, evidence
Phases 1 and 7 are where most engagements succeed or fail — clarity at the start, clarity at the end.

📋 Scoping — what to nail down

  • Targets: exact hosts, URLs, IP ranges, services. No "all of *.example.com".
  • Out-of-scope: third parties, prod databases, social engineering staff, DoS
  • Time window: testing hours, freeze on releases, blackout dates
  • Source IPs: tester's expected traffic origin (so blue team can correlate)
  • Data handling: if data is accessed, what is allowed to leave?
  • Stop conditions: what triggers immediate halt & notify?
Get all of this in writing — signed by someone with authority to authorise it.
"Stop conditions" is the most-skipped item and the most important. PII access, prod outage, evidence of real intrusion — define what you do.

📝 Rules of Engagement (RoE)

The shorter document everyone signs

One page. Lists scope, allowed techniques, prohibited techniques, communication channels, escalation contacts, and signatures.

  • Tester must be able to point at the RoE on demand
  • Update it if anything material changes (and re-sign)
  • Keep a copy with the engagement evidence
RoE protects everyone — tester from accusations, defender from surprises, business from undefined risk.

⚖️ Tester ethics & safety rails

  • Do no harm — default to non-destructive proof-of-concept
  • Don't pivot beyond scope — even if a path opens up
  • Confidentiality — everything you saw belongs to the client
  • Stop & notify if you stumble on real intrusion or law-enforcement-relevant material
  • Don't store findings on personal infrastructure
Encourage learners to imagine their own data being on the system they're testing. That mindset shapes ethics.

📝 The report — what good looks like

Executive section

  • Scope & dates
  • Risk summary, business impact
  • Key recommendations
  • One-page heat-map

Technical section

  • One finding per page: title, CVSS, repro steps, evidence, remediation
  • Methodology & tools
  • Out-of-scope items & assumptions
  • Appendix: raw scan output
Best report I ever read let an engineer fix a finding in one sitting without asking the tester a question.
Reports are an artefact of professionalism. Sloppy reports = sloppy engagement, regardless of technical depth.

🔥 Common pitfalls

  • Tool-reported findings as the report — "did you actually try them?"
  • Compliance-only test — passes the audit, misses the threat
  • "Pen test as security strategy" — one a year is not a program
  • No retest — findings linger; no proof of fix
  • Findings without business framing — CVSS 9.8s that no one prioritises
Most pen-test "fail to deliver value" stories are explained by these.

🔧 Lab 5: Scoped mini pen-test

Run a small grey-box engagement against the lab target

  • Read the provided scope & RoE document
  • Recon: identify services, versions, accessible endpoints
  • Vulnerability analysis: pick one likely path
  • Exploit (lab target only) and capture evidence
  • Write a one-page finding using the report template
Duration: ~45 minutes · Pair: Yes
Constrain ambition — many learners get lost trying for a perfect chain. Capture one finding well, end on time.
