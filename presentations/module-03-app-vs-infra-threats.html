Two layers, one stack — and a cloud twist

🔗 Two layers, two threat surfaces

Application threats

Risks introduced by your code & its dependencies.

  • Injection (SQL, command, template)
  • Broken access control / IDOR
  • Auth & session flaws
  • Server-side request forgery (SSRF)
  • Vulnerable dependencies

Infrastructure threats

Risks in the platform the app runs on.

  • Unpatched OS / kernel
  • Open management ports (SSH, RDP)
  • Mis-scoped IAM / over-privileged identities
  • Mis-configured network / firewall
  • Plaintext secrets in env / images
Reality check: most real-world breaches chain at least one of each. App flaw → infra pivot → data.
Ask: in their last incident, which layer was the entry point and which was the data-loss layer? Often two different layers.

💻 Anatomy of an application threat

SQL injection — one minute

cursor.execute(
  "SELECT * FROM users WHERE email = '" + email + "'"
)

If email is ' OR 1=1 --, the query returns every user.

  • Root cause: mixing user-controlled data with code (string concatenation)
  • Fix: parameterised queries / prepared statements
  • Why developers own it: only the code knows what's "data" vs "code"
Use this as the canonical app-layer issue. We'll see it again in module 5 (OWASP Top 10) and lab 4.

🏗️ Anatomy of an infrastructure threat

S3 bucket left public — no app bug needed

resource "aws_s3_bucket" "data" { bucket = "prod-data" }

resource "aws_s3_bucket_public_access_block" "data" {
  bucket                  = aws_s3_bucket.data.id
  block_public_acls       = false   # ← the bug
  block_public_policy     = false   # ← the bug
  ignore_public_acls      = false
  restrict_public_buckets = false
}
  • Root cause: a configuration default or a hand-typed flag
  • Fix: policy-as-code rejects this in CI; default-deny at platform level
  • Why platform owns it: the app didn't run yet — the door is already open
Compare to the app slide: same impact (data loss), different stage and team. Drives home why "DevSecOps" needs ops, not just dev.

Constant across every column: identity and data are always your responsibility.
AWS's published model. The single biggest cloud failure mode is assuming AWS handles a layer it doesn't.

☁️ Cloud changes the threat model

  • Identity is the new perimeter — one leaked token > one leaked password
  • The control plane is internet-exposed — mis-set RBAC = mis-set firewall
  • Everything is an API — misconfigurations propagate at deploy speed
  • Multi-tenancy — "noisy neighbour" becomes "side-channel"
  • Ephemerality — the host you're investigating may already be gone
Pause on identity. Most cloud incidents start with credential or token compromise (Verizon DBIR consistently flags this).

🌐 AWS-specific threat classes

Identity (IAM)

  • Over-privileged roles & users
  • Long-lived access keys
  • Wildcards in policies (Action: *)
  • No MFA on root / privileged users

Network

  • Security group rules 0.0.0.0/0
  • Public IP on app tier
  • Missing VPC endpoints
  • SSH/RDP without Session Manager / bastion

Data & secrets

  • Public S3 buckets / pre-signed URL leaks
  • Plaintext credentials in env vars or AMI
  • Secrets in code/Lambda instead of Secrets Manager
  • Disabled CloudTrail / S3 access logs
These map to AWS Security Hub's recommendation categories. Helpful context for learners who'll see Security Hub findings later.

🔑 Identity is the perimeter

Three habits that close 80% of cloud-IAM risk

  • Least privilege — scope role assignments to the smallest resource group / resource that works
  • Just-in-time elevation — PIM for human admins, managed identities for workloads
  • No standing secrets — rotate, expire, and store in Key Vault
Default rule: if an action can be done with a managed identity, it should never be done with a stored credential.
Plug AWS Secrets Manager here — we'll see it again in Module 8 (CloudWatch).

✅ Module recap

Two layers: application threats live in your code & deps; infrastructure threats live in your platform & config.
Cloud twist: shared responsibility, identity-as-perimeter, control plane is on the internet.
Place the control where it can be reasoned about — not where it's most convenient to add.
Hand off to the hands-on — catching infra threats at IaC time.
