🗺️ Threat Maps & Attack Surface

From whiteboard sketch to actionable backlog

DevSecOps — Module 4 of 9

Threat modelling is the highest-leverage security activity per minute spent. Lab 4 is the practical anchor.

🎯 Module objectives

  • Define attack surface & trust boundary in concrete terms
  • Draw a Data-Flow Diagram (DFD) for a realistic system
  • Apply STRIDE to enumerate threats per element
  • Use attack trees to reason about adversary goals
  • Convert findings into prioritised, actionable backlog items
By the end, learners should run a 30-minute threat-model session for a service their team owns.

🧠 Why threat model?

  • Surfaces design-time bugs that no scanner can find
  • Forces explicit articulation of trust boundaries
  • Generates a backlog of mitigations — not a list of CVEs
  • Cheap: a whiteboard, an hour, three people
The four questions (Adam Shostack):
  1. What are we building?
  2. What can go wrong?
  3. What are we going to do about it?
  4. Did we do a good job?
Shostack's four-question frame is the simplest possible meeting agenda. If a team only adopts one practice from the day, it should be this.

📝 Vocabulary, kept simple

TermPlain meaning
AssetAnything you'd be sad to lose — data, money, reputation
ThreatA bad outcome (e.g. "user data leaked")
VulnerabilityA weakness that lets the threat happen
Attack vectorThe path / mechanism the attacker uses
Trust boundaryWhere data crosses between principals with different trust levels
Attack surfaceSum of every place input crosses a trust boundary

📊 Data-flow diagrams (DFDs)

Five symbols

  • External entity — rectangle
  • Process — circle
  • Data store — two parallel lines
  • Data flow — arrow
  • Trust boundary — dashed line

Drawing rule

Every arrow that crosses a dashed line is a place to ask "what could go wrong?"

That's it. The whole point.

Tools: whiteboard, draw.io, OWASP Threat Dragon, Microsoft Threat Modeling Tool. Pick whichever your team will actually use.
Whiteboard wins for first sessions. Tooling matters more for ongoing maintenance.

🎯 Worked example — e-commerce checkout

System

Browser → CDN → web app → orders DB · payments service → Stripe API

Trust boundaries

  • Internet → CDN
  • CDN → web app
  • Web app → payments service
  • Payments service → Stripe

Crossing flows to question

  • User auth credentials
  • Cart contents (price tampering?)
  • Internal RPC tokens
  • Payment tokens
Walk this example aloud. Show each crossing flow as a "what could go wrong?" prompt.

🛡️ STRIDE — six categories of "what could go wrong"

LetterThreatProperty it violates
SSpoofingAuthentication
TTamperingIntegrity
RRepudiationNon-repudiation
IInformation disclosureConfidentiality
DDenial of serviceAvailability
EElevation of privilegeAuthorisation
How to apply: for each element of the DFD, ask each STRIDE letter in turn. Capture only the threats that actually apply.
STRIDE's purpose is to be a comprehension aid. Don't enumerate threats that have nothing to do with the element.

🌲 Attack trees

Top-down complement to STRIDE

Goal at root ("attacker reads admin email"). Branches are sub-goals; leaves are concrete steps.

Read admin email
├ Phish admin
│ ├ Send credential-harvest link
│ └ Send malicious attachment
├ Steal session token
│ ├ XSS
│ └ Steal browser storage
└ Compromise mailbox infra
  ├ Take over Entra ID account
  └ Abuse mail-forwarding rule
Use STRIDE to find threats; use attack trees to plan mitigations. They complement, they don't compete.
Schneier-style attack trees. Useful for prioritisation — kill an entire branch with one mitigation.

📊 Prioritising threats

DREAD (legacy)

Damage, Reproducibility, Exploitability, Affected users, Discoverability. Subjective — use sparingly.

CVSS

Standard score for known vulns. Doesn't fit threats well, but good once a threat becomes a CVE.

Risk = likelihood × impact

Two-axis qualitative grid. Easy to defend in a meeting. Often enough.

Don't get stuck on the framework — pick one and ship the backlog.
Many teams stall in the prioritisation phase. The cure is a deadline, not a better framework.

🛡️ From threats to mitigations

STRIDEDefault mitigation classes
SpoofingAuthentication, MFA, mTLS, signed tokens
TamperingHashes, signatures, integrity checks, ACLs
RepudiationAudit logs, signed events, secure timestamping
Information disclosureEncryption (at-rest & in-transit), least-privilege reads
Denial of serviceRate limits, quotas, autoscaling, circuit breakers
Elevation of privilegeAuthorisation checks, sandboxing, isolation
A starting palette, not a recipe. Real mitigations always need domain context.

🔧 Lab 4: Build an attack map

Threat-model a sample three-tier app on Azure

  • Read the architecture brief in the lab folder
  • Sketch the DFD (whiteboard, draw.io, or Threat Dragon)
  • Apply STRIDE-per-element — capture at least one threat per element
  • Convert the top three threats into mitigation backlog items
  • Walk your map for one other pair — pen-test their assumptions
Duration: ~45 minutes · Pair: Yes
Pairs work well here. Encourage debate during the cross-pair walk-through.
