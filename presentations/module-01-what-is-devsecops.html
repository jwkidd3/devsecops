📝 A working definition

DevSecOps is…

The practice of embedding security responsibility, automation, and feedback into every stage of the software delivery lifecycle — so that secure software is the default outcome of the pipeline, not a final gate.

Three words: Culture · Automation · Feedback. Drop any of the three and you're back to "Sec" being a bottleneck.
Push back on the common misread that DevSecOps is "DevOps + a security tool." Tools without culture or feedback fail the same way DevOps fails when teams don't change ownership.

Highlight that DevSecOps didn't replace DevOps — it answered DevOps's accidental side-effect of leaving security behind.

⚠️ Why now? The business case

Cost of late fixes

  • Fix in design: ~1x
  • Fix in build: ~10x
  • Fix in test: ~100x
  • Fix in production: ~1000x

External pressure

  • Regulators (GDPR, HIPAA, PCI, SOC 2)
  • Customer security questionnaires
  • Cyber-insurance requirements
  • Supply-chain attacks (SolarWinds, Log4Shell)
The line that lands: "We can't fix our way out of a security backlog at deployment speed."
Use the cost ratio to justify shift-left investment. Numbers come from NIST and IBM studies — orders of magnitude, not exact figures.

🎯 Goals of a DevSecOps program

  • Shared ownership — every engineer owns the security of code they ship
  • Fast feedback — vulnerabilities surface in minutes, not weeks
  • Automation by default — humans review findings, machines find them
  • Continuous compliance — evidence is a by-product of the pipeline
  • Reduced blast radius — assume breach; contain it
Each goal is observable. Ask the room which their org most lacks today — it's usually shared ownership.

🔄 The DevSecOps lifecycle

Plan & code

  • Threat modelling
  • Secure-coding standards
  • IDE security plugins
  • Pre-commit secret scans

Build & test

  • SAST (static)
  • SCA (dependencies)
  • IaC scanning
  • Container image scan

Release & operate

  • DAST (dynamic)
  • Signed artefacts
  • Runtime monitoring
  • Incident playbooks
Rule of thumb: if a control only runs at one stage, it's an audit. If it runs at every stage, it's DevSecOps.
We'll spend day 2 on tools that fill these stages — ZAP for DAST, Metasploit for offensive validation, Azure Monitor for runtime, Jenkins for orchestration.

⬅️ Shift left — and right

Shift left

Move security earlier. Cheaper, faster, and developers see findings in tools they already use.

  • Threat models in design
  • Secrets blocked at commit
  • SAST in PR

Shift right

You can't catch every issue at build time. Detection and response in production close the loop.

  • Runtime telemetry
  • WAF / RASP
  • Chaos & red-team exercises
Bidirectional — either alone gives you a false sense of safety.

🔥 Failure modes

  • Tool-first thinking — buying ten scanners before defining who owns findings
  • The DevSecOps team — centralising security undoes the shared-ownership goal
  • Alert fatigue — high false-positive rates train developers to ignore reports
  • Blocking gates without budget — failing the build with no time to fix breeds bypasses
  • Compliance theatre — ticking boxes for an auditor while real risk goes unmanaged
Watch out: a "DevSecOps" rebrand of the security team is the most common anti-pattern.
Ask the room: which of these have they seen? It's a great icebreaker for the next module.

✅ Module recap

Definition: security embedded in every stage of delivery, by every role, with automation closing the feedback loop.
Why: late fixes cost orders of magnitude more — and customers, regulators, and insurers demand it.
How it fails: tool-first thinking, a centralised "DevSecOps team," alert fatigue, theatre.

🔧 Lab 1: Sandbox setup

Stand up the AWS Cloud9 environment we'll use for every lab

  • Create a personal Cloud9 environment (devsecops-lab-<your-name>) in the shared training account
